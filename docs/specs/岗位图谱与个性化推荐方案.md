## 岗位图谱与个性化推荐整体方案

> 目标：基于现有 JD 数据 + 用户软硬实力画像，构建「岗位晋升/转岗图谱」并给出个性化岗位/路径推荐。

---

## 一、数据现状与整体思路

- **已有数据**：
  - `job_position`：从 `a13*.xls` 导入的大量 JD（岗位名称、地点、薪资、公司、行业、JD 文本等）。
  - `UserProfile`：最高学历、教育背景、绩点、意向行业/岗位等硬条件。
  - `UserPreference`：软实力自评、MBTI / Holland / Big Five 等测试结果。
- **整体原则**：
  - **晋升关系**：主要用「规则 + 标准化岗位名称」从 DB 中推导，不依赖 AI，保证可控和可解释。
  - **可转岗关系**：用「规则/领域经验 + 相似度/AI 建议」生成候选，再人工审核后写入 DB。
  - **个性化推荐**：把“岗位”和“用户”都映射到同一套标签空间（硬技能/软技能/兴趣&人格），在此基础上做匹配和打分。

## 二、如何获得岗位之间的晋升关系

### 2.1 关键思路：岗位家族 + 级别

对每个岗位抽取两个维度：

- **岗位家族 (`job_family`)**：同一类岗位的集合。
  - 例：
    - “前端开发工程师 / 前端开发 / Web 前端” → `frontend_dev`
    - “产品经理 / 资深产品经理 / 产品负责人” → `product_manager`
- **级别 (`job_level`)**：在该家族内的资历层级。
  - 例（可根据中文关键字配置）：
    - `0`：实习 / 练习生
    - `1`：助理 / 初级 / 专员 / 工程师
    - `2`：中级 / 高级 / 资深 / Leader / 主管
    - `3`：经理 / 负责人 / Owner
    - `4`：总监 / Head / VP / CTO / CPO

在同一 `job_family` 内，按 `job_level` 从低到高排序，就形成一条自然的**晋升链路**。

### 2.2 实施步骤

1. **设计岗位家族映射规则**
   - 新建一个字典或配置表，例如：
     - `frontend_dev`：包含 `"前端"`, `"Web前端"`, `"前端开发"` 等关键字。
     - `backend_dev`、`product_manager`、`operation`、`designer` 等类似。
   - 对 `job_position.title`（必要时 + `industry`）执行规则匹配，得到 `job_family`。
   - 若规则无法决定，可选：用 AI 做一次离线“归类”建议，人工检查后落库。

2. **设计级别解析规则**
   - 针对中文 title 写一套正则/映射表，例如：
     - 包含 `"实习"` → `job_level = 0`
     - 包含 `"助理"、"初级"、"专员"、"工程师"` 且不含 “高级/资深/经理/总监” → `1`
     - 包含 `"中级"、"高级"、"资深"、"主管"` → `2`
     - 包含 `"经理"、"负责人"` → `3`
     - 包含 `"总监"、"VP"、"总裁"、"CXO"` → `4`
   - 写成一个后端工具方法，对所有 `job_position` 扫一遍，填充到新列 `job_level`。

3. **落库方式**
   - 在 `job_position` 表中新增：
     - `job_family VARCHAR(64)`
     - `job_level INT`
   - 或新增一张**标准节点表**（你现在已有 `career_node`）：
     - 每个 family + level 合成一个标准节点（例如 `product_manager` @ `level=1` → “产品助理”）。
     - 多个真实 JD 可以指向同一个标准节点。

4. **生成晋升路径**
   - 对每个 `job_family`：按 `job_level` 排序，合并到 5–7 个代表性的节点，写入 `career_node`。
   - 这就是你图谱上的“藤蔓主干线”。

> 小结：晋升链路基本不依赖 AI，主要是 **中文职位名标准化 + 规则拆级别**，很稳定。

## 三、如何获得岗位之间的可转岗关系

### 3.1 基线：规则 + 域知识

先手工/规则写一批明显的可转岗关系：

- 产品线：
  - `产品经理 → 运营经理 / 项目经理 / 数据分析师`
  - `高级产品经理 → 产品总监 / 创业合伙人`
- 运营线：
  - `运营经理 → 产品经理 / 市场经理`
  - `运营总监 → 事业部负责人 / 产品总监`
- 技术线：
  - `高级开发 → 技术经理 / 架构师 / 技术合伙人`
- 设计线：
  - `高级设计师 → 设计主管 / 产品经理`

把这些关系写入 `career_transfer_option`（或对应表），即可支撑最基本的可转岗图谱。

### 3.2 用相似度 + AI 扩展候选

在基线之上，用“语义相似度 + 标签”找更多横向候选，再用 AI 只负责“解释文案”：

1. **向量相似度筛候选**
   - 为每条 `job_position` 生成嵌入向量（如 `title + description`）。
   - 对每个标准节点（或 family+level 组合）：
     - 在同城市/同大类行业/相近薪资的岗位中找 **top-K 相似岗位**。
     - 过滤：
       - 同一 `job_family` 且 `|level_diff| ≤ 1` → 视作平级/近级转岗候选。
       - 不同 family 但技能标签高度匹配（见下一节）→ 跨职能候选。

2. **AI 生成转岗说明（可选）**
   - 对于已选定的 `(来源岗位, 目标岗位)`：
     - 把两边的 JD 和标签给 AI，请它输出一句话说明，如：
       - “从用户需求洞察转向业务运营，适合沟通和跨部门协作强的人。”
       - “从代码实现转向技术团队管理与规划。”
   - 仅用于生成 `target_desc` 文案，**不要用 AI 决定连不连边**。

3. **写入 DB 前人工审核**
   - 建议流程：
     - 脚本把候选转岗关系导出为 JSON 或表格（含来源节点、目标节点、AI 文案）。
     - 人工审核勾选后，再写入 `career_transfer_option`。

> 小结：可转岗关系由 “规则/相似度” 决定结构，AI 只负责“润色解释文案”，结果落库后给前端用。

## 四、用户软硬实力如何映射到岗位（标签体系）

### 4.1 统一的标签空间

为方便匹配，需要在“用户侧”和“岗位侧”使用同一套标签体系，大致包含：

- **硬技能标签**（hard_skills）：
  - 技术栈：Java、Spring、Vue、Python、Excel、SQL、数据分析、Photoshop、Figma...
  - 证书：CET-4/CET-6、计算机二级、教师资格证等。
- **软技能标签**（soft_skills）：
  - 沟通表达、团队协作、领导力、项目管理、结构化思维、数据敏感度、抗压、同理心等。
- **兴趣 / 人格标签**（traits/interests）：
  - 对应 MBTI / Holland / Big Five 的维度：外向/内向、创造/执行、研究/社交/企业型等。

### 4.2 岗位侧打标签

1. **基于关键词/规则打硬技能标签**
   - 在 `job_position.description` 中用关键词匹配：
     - 出现 \"Java\"、\"Spring\" → `hard_skills` 包含 `java`, `spring`。
     - 出现 \"数据分析\"、\"SQL\"、\"Excel\" → 包含 `data_analysis`, `sql`, `excel`。

2. **基于关键词/规则打软技能标签**
   - 描述中有：
     - “沟通”、“跨部门协作”、“汇报” → `communication`, `teamwork`。
     - “项目推进”、“进度管理”、“协调资源” → `project_management`。

3. **可选：AI 提取标签**
   - 用一个离线脚本把 JD 文本喂给模型，让它输出：
     - `top 3 硬技能`, `top 3 软技能`, `适合的人格特质`，目标仍然是落到上面的标签空间。
   - 对 AI 结果做抽样检查后写入 DB（如 `job_hard_skills`, `job_soft_skills`, `job_traits` JSON 字段）。

### 4.3 用户侧打标签

1. **硬技能**
   - 来自简历里的：专业、课程、证书、项目经历；
   - 在前端收集时尽量结构化（多选技能 / 自填技能），后端存为 `user_hard_skills` 列表。

2. **软技能、自评与测评结果**
   - 你已有的软实力问卷题目本身就可以映射到 `soft_skills`：
     - “数据分析能力”题目 → `data_analysis` + 分值；
     - “沟通与表达”题目 → `communication` + 分值；
   - MBTI / Holland / Big Five：
     - 例如 ENFJ → `extrovert`, `empathy`, `leadership` 等 traits；
     - Holland RIASEC 中 S/E 型 → 适合社交、企业型工作。

> 小结：岗位和用户都变成「硬技能向量 + 软技能向量 + 性格/兴趣向量」，后续匹配就有了基础。

## 五、个性化推荐打分思路

### 5.1 过滤 + 打分

1. **先做硬过滤**：
   - 地点：用户接受城市集合 vs 岗位地点；
   - 学历：用户最高学历 vs 岗位最低学历要求（若有）；
   - 薪资：岗位薪资区间 vs 用户期望区间；
   - 岗位类型/行业：是否与用户意向一致。

2. **再做多维打分**（示例权重，可调整）：

- `S_hard`：硬技能匹配度（0–1）
  - 计算方式：`交集 / 岗位硬技能数量` 或余弦相似度。
- `S_soft`：软技能 & 人格匹配度（0–1）
  - 岗位的软技能 / traits 向量 vs 用户的 soft skills / traits 向量相似度。
- `S_interest`：兴趣 / 职业倾向匹配度（0–1）
  - 如 Holland / MBTI 维度的匹配。

综合：

\\[ S_{total} = 0.4 \\cdot S_{hard} + 0.3 \\cdot S_{soft} + 0.3 \\cdot S_{interest} \\]

并可根据业务调整权重。

### 5.2 在图谱上的推荐形式

- **当前起点岗位推荐**：
  - 在所有节点中，找出对当前用户 `S_total` 高，且 level 与用户当前能力/经历匹配的几个节点，作为“适合你当前起点的岗位”。
- **晋升路径推荐**：
  - 从起点顺着该线路的晋升链路向上，看哪几个节点 `S_total` 也比较高，展示为“未来 2–3 个可努力的目标岗位”。
- **转岗路径推荐**：
  - 对每个可转岗节点的 `career_transfer_option`，只展示 `S_total` 高于某阈值的目标，而不是全部展示。

## 六、AI 应该放在哪些环节

### 6.1 适合用 AI 的地方

- 帮忙**归类岗位家族**（生成建议，再人工定稿）。
- 从 JD 描述中**自动提取硬技能/软技能/人格标签**，减少人工标注工作量。
- 对已决定的“可转岗对”生成**自然语言说明**（`target_desc`）。
- 对一批岗位名生成初版的**职业线 + 升岗/转岗建议 JSON**，供你审阅后写入 `career_*` 表。

### 6.2 不建议完全交给 AI 的地方

- 直接让 AI 在线生成整张图谱并直接给用户看（难以控制、难以审计）。
- 不经审核就根据 AI 的建议自动写库，可能会产生明显不合理的路径。

> 推荐模式：**AI 只做“建议 + 文案”，结构与最终数据都写入 DB 后由接口统一输出，前端不直接依赖 AI。**

## 七、推荐的落地顺序（Checklist）

1. **岗位标准化**
   - [ ] 设计并实现 `job_family` 规则（或配置表），批量填充到 `job_position`。
   - [ ] 设计并实现 `job_level` 解析规则，批量填充。

2. **图谱结构**
   - [ ] 基于 family + level 在 DB 里维护标准节点和晋升路线（可用现有 `career_track`/`career_node`）。
   - [ ] 通过规则先写一版“显而易见”的可转岗关系到 `career_transfer_option`。

3. **AI 辅助扩展**
   - [ ] 使用 `suggest_career_map.py` 或类似脚本，从 xls 抽岗位名，并调用 OpenAI 生成职业线/转岗候选 JSON。
   - [ ] 人工审阅 JSON，决定哪些线路/转岗关系写入 DB。

4. **用户与岗位打标签**
   - [ ] 设计统一的技能 / 特质标签字典。
   - [ ] 岗位侧：在批处理脚本或定时任务中基于 JD 文本打标签（可选 AI）。
   - [ ] 用户侧：在已有问卷/简历上增加/映射到同一套标签，并存入 DB。

5. **推荐服务**
   - [ ] 在后端实现一个 `computeMatchScore(user, job)` 的简单打分函数（按 hard/soft/interest）。
   - [ ] 在接口层增加“推荐岗位列表 / 推荐图谱路径”的 API，前端只负责展示。

按这个文档一步步落地，就可以做到：
- 晋升/转岗关系**结构上由 DB 控制**，
- **AI 只做建议和文案**，
- 最终基于用户软硬实力做出**个性化的岗位和路径推荐**。

## 八、本地大模型部署与集成建议

> 目标：在不依赖外部 API 的前提下，在本地部署大模型，用于标签抽取、推荐解释和轻量决策，而不是完全端到端“算推荐结果”。

### 8.1 模型与部署方式建议

- **模型大小与类型**（中文 + 结构化输出友好）：
  - 推荐优先：`Qwen2.5-7B-Instruct` / `Qwen2.5-14B-Instruct`（中文能力强，适合解释与工具调用）。
  - 如偏重代码逻辑：可考虑 `DeepSeek-Coder` 或 `DeepSeek-V2` 系列。
- **部署方式**：
  - 开发机 / Mac：`ollama` + Qwen7B（入门最快）。
  - 服务器 / GPU：`vllm` 或 `text-generation-inference` 加载 Qwen/Llama，走 HTTP/GRPC 推理。
  - 轻量 CPU：`llama.cpp` / `gpt4all` 量化模型。

总体建议：**先用 ollama + Qwen2.5-7B 在本地起一个服务，验证 prompt 与 JSON 输出，再考虑上服务器与微调。**

### 8.2 大模型在方案中的角色定位

大模型主要承担两个角色：

1. **离线数据处理工**（批处理脚本）：
   - 从 JD 抽取：`job_family` 候选、`hard_skills`、`soft_skills`、`traits` 等标签。
   - 从用户测评结果抽取：人格/软实力的自然语言说明与标签。
   - 归一化岗位名称（标准 title）与家族。
2. **在线“解释器 + 轻量决策器”**：
   - 输入：规则系统算出的匹配度、用户画像、候选岗位/路径。
   - 输出：排序建议、小范围 re-rank 和“为什么推荐这些”的结构化解释。

> **不建议**：让大模型直接端到端给出最终推荐列表并绕过规则/打分逻辑，否则难以控制与解释。

### 8.3 微调任务设计（而非训练“推荐模型”）

你更需要的是让模型在以下小任务上更稳定：

1. **分类与标注**（多标签任务）：
   - 输入：`job_position.title + description`。
   - 输出：
     ```json
     {
       "job_family": "product_manager",
       "job_level": 2,
       "hard_skills": ["需求分析", "PRD撰写", "数据分析"],
       "soft_skills": ["沟通协作", "同理心"],
       "traits": ["偏外向", "用户导向"]
     }
     ```
2. **映射与解释**：
   - 输入：用户测评结果 + 用户标签 + 岗位标签。
   - 输出：简短解释“该岗位为何适合/不适合该用户”。
3. **路径解释**：
   - 输入：图谱上的路径（当前节点 → 1～3 个下一步）+ 匹配分。
   - 输出：对每条路径的一段人话说明。

#### 8.3.1 微调技术路线

- 使用 **LoRA / QLoRA** 进行指令微调即可：
  - 工具如 `LLaMA-Factory`、`xtuner`、Qwen 官方工具均支持 LoRA。
  - 数据来源：
    - 手工标注少量 JD → 标签、level、family；
    - 手写若干高质量推荐解释文本，作为监督样本。
- 重点是让模型**习惯输出 JSON 结构**，而非自然段落。

### 8.4 引入本地大模型后，方案的调整点

在前面的推荐方案基础上，本地大模型主要增强两块：

1. **标签与匹配模块的“LLM 版本”**：
   - 使用离线脚本 + 本地 LLM 自动为岗位打标签（family/level/skills/traits），写入 DB。
   - 用户侧同理，从测评结果中自动提炼软实力/人格标签并落库。
   - 在线匹配时仍然只用 **DB 中的标签与数值**，不每次实时调用 LLM。
2. **推荐解释与可视逻辑输出**：
   - 在线推荐服务先用规则模型算出候选岗位/路径 + `S_hard/S_soft/S_interest` 分数。
   - 把这些结构化信息喂给本地 LLM，让它输出：
     ```json
     {
       "sorted_job_ids": ["p3", "o3", "t3"],
       "reasons": {
         "p3": {
           "hard_fit": "...",
           "soft_fit": "...",
           "path_reason": "..."
         }
       },
       "recommended_path": [
         { "from": "p2", "to": "p3", "reason": "..." }
       ],
       "steps": [
         { "step": "filter", "description": "...", "kept_job_ids": [...] },
         { "step": "score", "description": "...", "scores": { "p3": 0.86 } },
         { "step": "graph_path", "description": "...", "path": ["p2", "p3", "o3"] }
       ]
     }
     ```
   - 后端只需把这些结构映射到前端的 UI（路径高亮、hover 提示、侧边“决策步骤”），就能实现“可视逻辑”。

### 8.5 “可视逻辑”输出的实践要点

1. **约束大模型输出为 JSON**：
   - 通过 prompt 严格要求输出 JSON，对异常输出做兜底解析。
   - JSON 中包含：排序、路径、每一步推理的描述（`steps`）和每个推荐项的理由（`reasons`）。
2. **前端可视化**：
   - 图谱节点 hover：展示拆解后的匹配分（硬/软/兴趣）+ 一句话解释。
   - 边上或节点下方显示路径推荐理由（`recommended_path.reason`）。
   - 侧边栏展示 `steps`，让用户和开发者都看到“过滤 → 打分 → 选路径”的过程。
3. **保持规则与数据主导**：
   - 所有硬约束（城市/学历/薪资等）与基础打分仍由规则模型控制；
   - 大模型只在候选集合上做微调和解释，以保证系统可控、可调试。

## 九、与现有方案的结合方式总结

- **不变的核心**：
  - 图谱结构存 DB（`career_*` 表），前端调 `GET /job/career-map`。
  - 推荐打分仍以「规则 + 标签匹配」为主，数值逻辑你完全掌控。
- **新增加的本地大模型能力**：
  - 离线：辅助岗位/用户打标签、生产职业线/转岗建议初稿。
  - 在线：在候选结果之上做**解释生成 + 小范围排序 + 决策步骤 JSON 输出**。
- **最终效果**：
  - 推荐结果可解释（有数值拆解、有自然语言理由、有步骤视图）。
  - 即便大模型不可用，系统仍能靠规则/图谱跑起来，只是少了“会说话”的那一层。

## 十、微调与评测：比赛可展示的量化指标

> 目标：在**没有用户行为数据（点击/投递）**的情况下，构造一套可重复的评测任务和指标，用来展示“微调前后模型在该领域的能力提升”。

### 10.1 选择可量化的子任务

围绕“理解岗位”和“支撑图谱”的能力，拆成 2–3 个简单任务：

- **任务 A：岗位家族分类 (`job_family`)**
  - 标签例：`frontend_dev`, `backend_dev`, `product_manager`, `operation`, `designer`, `data_analyst` 等。
  - 衡量：给定 JD 文本，模型能否正确判断岗位属于哪类。
- **任务 B：级别判断 (`job_level`)**
  - 标签例：0=实习，1=初级/专员，2=中高级/主管，3=经理/负责人，4=总监/VP/CTO…  
  - 衡量：模型是否理解岗位的“资历层级”，直接关系到晋升链路质量。
- **任务 C：技能标签抽取 (`hard_skills` / `soft_skills`)**
  - 例如从 JD 中抽取 3–5 个硬技能标签（Java/SQL/Excel/数据分析…）。  
  - 可做成多标签分类任务，评估 micro/macro F1。

> 这些任务都只依赖 JD，本身就是对“模型理解岗位能力”的量化刻画，适合比赛 PPT 展示。

### 10.2 构建小型评测集（gold set）

你没有大规模标注数据，但**比赛不需要几万条，只要几十～几百条高质量样本**即可：

1. **规则+脚本先产一版伪标签**：
   - 用关键字/正则：
     - title 里出现“前端/前端开发/Web 前端”→ `job_family = frontend_dev`；
     - 出现“实习”→ `job_level = 0`，出现“高级/资深”→ `2`，出现“经理/负责人”→ `3` 等；
     - 在 description 里提到的技能关键字 → 初步的 `hard_skills`/`soft_skills`。
2. **人工抽样修正**：
   - 从中抽取 **100～300 条**覆盖不同家族/级别的样本；
   - 人工在 Excel / JSON 中修正 family/level/skills → 形成统一的 `evaluation_set.json`。
3. **数据示例结构**：
   ```json
   {
     "job_id": 123,
     "title": "高级产品经理",
     "description": "...",
     "gold_family": "product_manager",
     "gold_level": 2,
     "gold_hard_skills": ["需求分析", "PRD撰写", "数据分析"],
     "gold_soft_skills": ["沟通协作", "跨部门协调"]
   }
   ```

这份 gold set 既可做 baseline 评测，也可比较微调后的效果。

### 10.3 微调前后对比指标

在同一评测集上，对比“规则/原模型” vs “微调后模型”：

- **任务 A（family）**：
  - 指标：`accuracy_family_before` vs `accuracy_family_after`。
  - 示例：`job_family` 准确率从 72% → 88%。
- **任务 B（level）**：
  - 指标：`macro-F1_level_before` vs `macro-F1_level_after`。
  - 示例：宏平均 F1 从 0.65 → 0.82。
- **任务 C（skills）**：
  - 指标：`hard_skills` micro/macro F1。
  - 示例：micro-F1 从 0.58 → 0.74。

这三条曲线/表格，就是比赛中“微调前后能力提升”的**硬证据**。

### 10.4 在没有行为数据时如何证明推荐价值

你缺少用户点击/投递数据，无法算 CTR/HR@K，但可以用两类间接证据：

1. **结构质量指标**：
   - 晋升链路单调性：图谱中 level 严格递增的边占比；
   - 可转岗合法性：满足“家族差距不过大 + 薪资无明显断崖”的边占比；
   - 这些可以统计为简单百分比（“合法边占全部边的 93%”）。
2. **小规模人工推荐评估**：
   - 构造 20～30 个“用户画像 + 推荐列表”的样本：
     - 一份来自纯规则系统；
     - 一份来自“规则 + 微调模型 re-rank/解释”的组合系统。
   - 请 2～3 位评审（你、队友、导师）盲评每个样本的“合理性/解释性”打 1–5 分；
   - 统计平均分与方差，对比前后：
     - 例如：平均合理性评分从 3.2 分 → 4.1 分。

在比赛材料里，可以这样陈述：

- “在 150 条人工标注 JD 上，我们的岗位家族分类准确率从 72% 提升到 88%，级别判断宏平均 F1 从 0.65 提升到 0.82，硬技能标签 micro-F1 从 0.58 提升到 0.74。”
- “在 30 个模拟用户画像样本上，我们请 3 位评审对推荐合理性打分，平均从 3.2 分提升到 4.1 分。”

这既显示了**模型理解岗位的能力提升**，又体现了**个性化推荐质量的主观提升**。

### 10.5 比赛用行动清单（实操路径）

1. **定义评测任务**  
   - [ ] 选定：`job_family` 分类、`job_level` 分类、`hard_skills` 多标签抽取 三个任务。
2. **构建评测集**  
   - [ ] 用规则在 `job_position` 上产初始伪标签；  
   - [ ] 抽样 100～300 条人工修正 → `evaluation_set.json`。
3. **跑 baseline**  
   - [ ] 用“规则 + 未微调模型”在评测集上预测，计算 accuracy / F1 作为 baseline。
4. **微调本地模型**  
   - [ ] 选 Qwen2.5-7B-Instruct（或类似）+ LoRA；  
   - [ ] 用“JD→标签”的示例做 SFT（见 8.3 节）。  
5. **评估微调效果**  
   - [ ] 在同一评测集上跑微调后模型，计算新指标；  
   - [ ] 对比 baseline，画表/图用于 PPT。  
6. **小型人工推荐评估**  
   - [ ] 构造 20～30 个“用户画像 + 推荐列表”样本；  
   - [ ] 让 2～3 人对“规则系统 vs 规则+LLM 系统”盲评；  
   - [ ] 汇总平均分与案例，放入汇报材料。

按这个章节执行，你就能在比赛中给出一套：  
**有任务定义、有评测集、有 baseline 对比、有主观评估** 的完整量化故事。
